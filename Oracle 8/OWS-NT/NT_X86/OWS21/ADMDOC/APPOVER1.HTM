Invoking an Application

A URL contains a virtual path that specifies the Web page to be returned. Whether this path specifies a static page, a WRB application, or a CGI application is determined by how the Listener is configured. See Using the Oracle WebServer Manager for details on configuring the Listener. This page describes how a URL that specifies either a WRB or a CGI application works. Except as noted, this information applies to applications written in any supported language and to custom-written WRB cartridges.

Note: This section is written as though you were passing parameters to procedures in a language such as PL/SQL. If you are using an object-oriented language like Java, you can use the same approach to apply instance variables to classes.

How the URL is Interpreted

The URL is examined and stored in the CGI variables shown in the table below.

Note: CGI environment variables are used regardless of whether the application is based on the WRB or CGI.

Environment Var.
Explanation
script_name
This is the virtual path that identifies the application to be executed. If the application is for the PL/SQL Agent, this includes the DCD.
path_info
This is the procedure that constitutes the main body of the application.
query_string
These are values for the input parameters, if any, of the procedure identified in path_info. This can also be passed as part of the HTTP header (see Request Methods for Passing Parameters)
request_method
If the parameter values are sent in the URL, this is set to GET. If they are sent in the HTTP header, this is set to POST. If no parameter values are passed to the procedure, this is undefined. For more information, see Request Methods for Passing Parameters and Example of PL/SQL Agent Parameter Passing Using GET

Note: HTML variables are associated with parameters by name. Therefore, you must use the same names for matching HTML variables and parameters.

Request Methods for Passing Parameters

There are two techniques, called request methods, for getting parameter values from a Web page to a procedure. They are as follows:

  • GET. In this method, the parameter values are appended to the URL. See Example of PL/SQL Agent Parameter Passing Using GET.
  • POST. In this method, the parameter values are passed as part of the HTTP header that accompanies every Web page request. See Example of PL/SQL Agent Parameter Passing Using POST.
Use GET when the parameter values are determined simply by which link the user selects. Since the URL is contained in the text of the link, you can simply include the parameter values in that code. The disadvantage of GET is that the entire string of name-value pairs must fit into a single environment variable. The maximum size of the environment variable is controlled by the Operating System. Nonetheless, on most systems it is long enough for a fairly high number of parameters.

POST is recommended when the parameter values are based on how the user fills in an HTML form. In HTML forms, you specify the request method as part of the form tag syntax. POST is recommended because it does not impose the length limitations of GET. Often you can also use POST for data generated by Java applets or browser plug-ins.

It is transparent to your application whether GET or POST is used because WebServer manages the parameter passing itself. You need not worry about parsing a CGI environment variable from your application to extract the query string. If you want to know see which request method was used for a given request, check the REQUEST_METHOD environment variable.

For either request method, the parameters are passed as name/value pairs in the following format:

parametername1=value1&parametername2=value2&parametername3=value3...

For an object-oriented language like Java, this would be:

class=instancevariable&class=instancevariable&class=instancevariable...
